# Plan.md

## 1. 기획 변경 요약

이번 개편에서 기존 전략을 다음과 같이 수정한다.

- 철회: `/line`, `/grab` 엔드포인트 분리 기반 멀티 미니앱 전략
- 확정: **LINE 전용 단일 미니앱 전략**
- 목표: 엔드포인트/플랫폼 복잡도를 줄이고, LINE 사용자 경험과 운영 안정성에 집중

즉, 제품은 LINE 컨테이너 환경 최적화를 최우선으로 하며, 확장보다 완성도를 먼저 달성한다.

---

## 2. 제품 목표 (LINE 전용)

1. LINE 유입 사용자가 즉시 사용할 수 있는 단일 진입 UX 제공
2. 서버 사이드 검증 중심의 안전한 인증/세션 구조 확보
3. 운영 관점에서 추적 가능한 로그/오류 관리 체계 구축
4. 추후 기능 확장(예약/주문/티켓/멤버십 등)을 고려한 구조 유지

---

## 3. 정보 구조(IA)

### 3.1 루트 `/` (단일 메인)
역할:
- LINE 미니앱의 메인 진입/홈
- 사용자 상태 로딩 및 주요 액션 노출

구성:
- 헤더: 서비스명, 상태 표시
- 본문: 핵심 기능 카드(서비스 도메인에 맞게 확장 가능)
- 상태 영역: 인증 상태, 오류 안내, 재시도 액션
- 푸터: 문의/약관/개인정보 처리 고지

### 3.2 오류/상태 화면
- 인증 실패
- 권한 부족
- 네트워크 오류
- 서버 점검

원칙:
- 실패 원인과 다음 행동(재시도/문의)을 명확히 제공

## 4. 라우팅/엔드포인트 정책 (최종)

### 4.1 페이지 라우트
- `GET /` : LINE 미니앱 메인

### 4.2 API 라우트 (MVP)
- `GET /api/health`

- `POST /api/entry-log`

### 4.3 제거 대상
- `/grab` 페이지/정책/SDK 연동 계획
- Grab 검증 API 및 관련 운영 항목
- 멀티 플랫폼 분기 로직

## 5. 기술 결정

### 5.1 프론트/런타임
- **Next.js (App Router) 고정**
- React Server Components + Client Components 혼합

### 5.2 인증/보안
- LINE 컨텍스트 검증을 서버 우선 처리
- 세션 토큰 수명/갱신 정책 명시
- 요청 단위 로깅(성공/실패 코드 포함)

### 5.3 데이터
- PostgreSQL + Prisma 권장
- 핵심 테이블(초안):
  - `users`
  - `line_sessions`
  - `entry_logs`
  - `error_logs`

### 5.4 운영
- Vercel 기준 배포
- dev/stg/prod 환경 분리
- 환경별 LINE 채널 설정 및 시크릿 분리

## 6. 기능 요구사항 (MVP)

1. LINE 컨테이너에서 `/` 진입 시 앱 초기화 성공
2. LINE 컨텍스트 검증 성공 시 사용자 세션 생성/갱신
3. 검증 실패 시 명확한 실패 이유와 재시도 UX 노출
4. 진입 이벤트/오류 이벤트 서버 적재
5. 기본 헬스체크 엔드포인트 운영

---

## 7. 비기능 요구사항

- 성능: 초기 주요 UI 표시 3초 이내 목표(모바일 네트워크 기준)
- 보안: 시크릿/토큰 안전 저장, 재사용 공격 방지
- 안정성: API 실패 시 graceful fallback 제공
- 관측성: 성공률/실패율/오류코드 기반 모니터링
- 유지보수성: 도메인 기능 추가 시 기존 인증 축을 재사용 가능해야 함

## 9. QA 체크리스트

- LINE 환경에서 `/` 진입이 안정적으로 동작하는가?
- `line/verify` 실패 시 메시지와 재시도 동선이 명확한가?
- 네트워크 단절/지연 상황에서 UX가 깨지지 않는가?
- `entry-log`가 정상 적재되고 필드 누락이 없는가?
- 운영자가 오류 상황을 로그로 재현 가능한가?

## 11. 결론

- 본 프로젝트는 **LINE 전용 단일 미니앱**으로 추진한다.
- Next.js(App Router)를 기반으로 서버 사이드 검증 중심 구조를 채택한다.
- 멀티 플랫폼(Grab 포함) 전략은 현재 범위에서 제외하며, 이후 필요 시 별도 프로젝트로 검토한다.
